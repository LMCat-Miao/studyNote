# 一、为什么 JavaScript 需要事件循环？

首先回顾一个核心：

## JavaScript 是单线程语言

什么意思？

就是：

> 同一时间只能执行一个任务。

例如：

```
console.log(1);

console.log(2);

console.log(3);
```

执行：

```
Call Stack

console.log(1)
↓
console.log(2)
↓
console.log(3)
```

输出：

```
1
2
3
```

很简单。

------

但是问题来了：

如果遇到耗时任务：

例如：

```
console.log("开始");

setTimeout(()=>{
    console.log("定时器");
},2000);


console.log("结束");
```

如果 JavaScript 等 2 秒：

那么页面是不是卡死？

用户点击不了按钮。

动画停止。

所以浏览器设计了：

> JavaScript 执行耗时任务时，把任务交给浏览器，自己继续执行。

这就是：

**事件循环机制。**

------

# 二、事件循环整体模型（必须背）

你可以先记住这一张图：

```
             JavaScript Runtime


                 Call Stack
                     |
                     |
                     ↓


              Web APIs
       (浏览器提供的能力)

     setTimeout
     ajax
     DOM事件


                     |
                     |
                     ↓


        Task Queue(任务队列)


        宏任务队列
        ----------------
        setTimeout
        setInterval
        DOM事件
        ajax回调


        微任务队列
        ----------------
        Promise.then
        MutationObserver


                     |
                     |
                     ↓


               Event Loop
```

核心流程：

```
执行同步代码

↓

遇到异步任务

↓

交给浏览器

↓

异步完成

↓

放入任务队列

↓

Event Loop检查Call Stack

↓

执行任务
```

# 三、什么是宏任务？

英文：

Macro Task

也叫：

Task

常见宏任务：

## 1. setTimeout

```
setTimeout(()=>{
    console.log("timer")
},0)
```

------

## 2. setInterval

```
setInterval(()=>{
    
},1000)
```

------

## 3. DOM事件

例如：

```
button.onclick=function(){

}
```

------

## 4. ajax回调

例如：

```
fetch('/user')
.then()
```

注意：

fetch本身不是宏任务。

但是：

网络请求完成后的回调属于任务。

------

# 四、什么是微任务？

英文：

Micro Task

常见微任务：

## 1. Promise.then

例如：

```
Promise.resolve()
.then(()=>{
    console.log("then")
})
```

------

## 2. async/await

例如：

```
async function test(){

    await Promise.resolve();

    console.log("hello");

}
```

await后面的代码：

本质也是微任务。

------

## 3. queueMicrotask

```
queueMicrotask(()=>{

})
```

------

# 五、最重要规则

面试必须会：

> 当前宏任务执行完以后，会优先清空所有微任务，然后执行下一个宏任务。

也就是：

```
同步代码

↓

微任务

↓

宏任务

↓

微任务

↓

宏任务
```

------

# 六、第一个事件循环题

看：

```
console.log(1);


setTimeout(()=>{
    console.log(2);
},0);


Promise.resolve()
.then(()=>{
    console.log(3);
});


console.log(4);
```

问：

输出什么？

------

## 第一步：执行同步代码

执行：

```
console.log(1)
```

输出：

```
1
```

遇到：

```
setTimeout
```

放入：

宏任务队列

```
宏任务:

[ timer ]
```

遇到：

```
Promise.then
```

放入：

微任务队列

```
微任务:

[ then ]
```

继续：

```
console.log(4)
```

输出：

```
4
```

现在：

同步代码结束。

------

## 第二步：执行微任务

微任务：

```
then
```

输出：

```
3
```

------

## 第三步：执行宏任务

timer执行：

输出：

```
2
```

最终：

```
1
4
3
2
```

# 七、为什么 Promise 比 setTimeout 快？

很多新人误解：

> setTimeout 0秒是不是马上执行？

不是。

比如：

```
setTimeout(()=>{
 console.log("timer")
},0)
```

意思：

> 最早下一轮事件循环执行。

而 Promise：

```
Promise.resolve()
.then(()=>{
 console.log("promise")
})
```

进入：

微任务队列。

规则：

当前代码结束：

先清空微任务。

所以：

```
Promise
先执行

setTimeout
后执行
```
